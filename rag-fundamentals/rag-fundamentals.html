RAG Fundamentals

Embeddings from scratch → retrieval → grounded generation
01

What is RAG, and why

An LLM only knows what was baked into it during training — nothing about your private docs, nothing newer than its training cutoff. Retrieval-Augmented Generation (RAG) fixes this without retraining: at query time, fetch the most relevant pieces of your data and stuff them into the prompt, so the model answers grounded in that context instead of guessing from memory.

The core pipeline

docs → chunks → embeddings → vector store → similarity search → augmented prompt → generation

We'll build every piece from scratch with raw numpy first, then show the same pipeline wired up with LangChain.

02

Embeddings from scratch

A text embedding is a fixed-length vector of floats that captures the meaning of a piece of text. Semantically similar text ends up as vectors pointing in similar directions. 03

Similarity — cosine similarity by hand

The formula

cos(a, b) = (a · b) / (‖a‖ ‖b‖) — the dot product of two vectors, divided by the product of their lengths. Result ranges from -1 (opposite) to 1 (identical direction), and it ignores vector magnitude, only the angle matters.

1.0 → same meaning. 0 → unrelated. This is the standard measure for comparing embeddings, and it's exactly what the stepper above just animated.

04

Chunking documents

Real documents are too long to embed as one blob and too long to stuff whole into a prompt. We split them into chunks — small enough to embed precisely, large enough to keep context coherent.

Chunk size is a tradeoff

Too small → loses surrounding context, a chunk about "the deploy tool" might lose which tool it's talking about. Too large → drowns out the one relevant sentence and wastes prompt budget on irrelevant text. Fixed-size splitting with a small overlap (so a sentence at a chunk boundary isn't cut mid-thought) is the simplest starting point.

05

Building the in-memory vector store

Embed every chunk once, stack the vectors into a matrix, keep the matrix in memory alongside the original chunk text. To query: embed the question, compare it against every row with cosine similarity, keep the top-k. A brute-force scan is all a small in-memory store needs — no index structure required until the corpus is large.

06

Retrieval → augmented prompt → generation

Given a user question: embed it, pull back the top-k most similar chunks, stuff them into a prompt template as context, then call the model. The model now answers grounded in the retrieved text instead of purely from its training data.
