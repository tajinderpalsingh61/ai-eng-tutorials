RAG in Production

Retrieval quality → evaluation → scaling → freshness
01

Why the toy pipeline breaks

RAG Fundamentals built a working pipeline — chunk → embed → cosine search → stuff context → generate — and proved it on 4 documents. It worked because 4 documents is a toy: every design shortcut that's invisible at that scale becomes a real, user-facing bug the moment the corpus is a real company's real documents. Four shortcuts in particular, each traceable to a specific design choice made for simplicity:

  • Pure vector search misses exact matches. The fundamentals pipeline retrieves by meaning alone. That's the right default for "what's your refund policy," where the words in the question don't need to literally appear in the answer — but it quietly fails for anything with an exact token that matters: an error code, a SKU, a person's name. Embeddings capture meaning, not keywords, so a query for an error code or SKU can score worse than a semantically related but wrong chunk.
  • Nobody measures whether retrieval is actually good. The fundamentals pipeline never asked "did we retrieve the right thing?" — it just retrieved something and moved on. Without a number to track, a change to chunk size or embedding model ships and quality regressions surface only when a user complains, not before.
  • O(n) cosine scan over every vector doesn't scale. Comparing the query against every stored vector one by one is fine at 4 documents and fine at 4,000 — it stops being fine somewhere past a few thousand chunks, where every query starts taking noticeably longer.
  • The index goes stale. The fundamentals pipeline embeds once and never revisits that embedding. The moment a source document changes, the stored vector is now describing something that no longer exists — and there's no signal anywhere that it happened.

Four levers fix each of these, one-to-one and in order: hybrid search, evaluation, ANN indexing, and freshness.

02

Hybrid search: BM25 + vector + Reciprocal Rank Fusion

A query for "E-4521" should retrieve the chunks that literally mention that error code. But an embedding model represents meaning — "error code" plus some digits doesn't have strong semantic neighbors, so a chunk about "checkout failing under load" can out-score it on cosine similarity even though it never mentions the code.

Fix: run two retrievers, fuse the rankings

  • Vector search alone struggles here because it's optimized for exactly one kind of question — "which chunk means roughly the same thing as this query" — and E-4521 isn't a meaning, it's a token. So bring in a second retriever that's good at exactly the thing vector search is bad at: matching tokens.
  • BM25 is that retriever — a lexical/keyword scorer descended from classic search-engine ranking.
  • The intuition: a chunk is relevant if it repeats the query's words, but not all words are equally informative. "the" appearing in every chunk tells you nothing; "E-4521" appearing in two chunks out of six is a strong signal.
  • That's what IDF (inverse document frequency) encodes — the rarer a term is across the whole corpus, the more weight a match on it gets.
BM25 score for one chunk, given a query
score = Σt IDF(t) · f · (k₁+1) f + k₁·(1 − b + b·L)
  • t — one term (word) from the query. The sum runs over every query term, one at a time.
  • IDF(t) — higher for rare terms across the corpus, near zero for common ones ("the," "and").
  • f — short for term frequency: how many times t shows up in this chunk.
  • k₁, b — tuning constants (typically 1.2-2.0 and 0.75) that control how quickly extra repeats stop mattering, and how much long chunks get penalized.
  • L — short for |chunk| / avg_len: this chunk's length relative to the corpus average (1.0 = exactly average length) — the length-normalization term, so a long chunk can't win purely by containing more words.
  • Now there are two ranked lists for the same query — one from BM25, one from cosine similarity — and they can disagree completely, because they're measuring different things.
  • Reciprocal Rank Fusion (RRF) combines them without ever comparing their raw scores directly, which matters because BM25 scores and cosine similarities live on completely different numeric scales (a BM25 score of 8 and a cosine similarity of 0.44 aren't comparable numbers).
  • RRF sidesteps that by only ever looking at rank position, never the raw score.
Reciprocal Rank Fusion
RRF(chunk) = Σlist 1k + ranklist(chunk)
  • ranklist(chunk) — this chunk's position in one retriever's ranked list (0 = top result).
  • k — a damping constant, commonly 60. Without it, the #1 result in one list would dominate the fused score regardless of how the other list ranked it; k flattens that gap so a chunk needs to rank well consistently, not just spike in one list.
  • Sum is taken over both retrievers' lists — a chunk absent from a list simply contributes 0 from that term.
Hybrid search + RRF — query: "what does error E-4521 mean"

Fused ranking correctly surfaces both E-4521 chunks first, ahead of the semantically-similar-but-wrong "checkout fails under load" chunk that pure vector search alone would have ranked #1.

03

Reranking

Hybrid search from section 02 is fast because both retrievers score every candidate in isolation — BM25 looks only at the chunk's own words, the embedding was computed once and stored long before this query ever arrived. That speed has a cost: neither retriever ever actually reads the query and the chunk side by side. RRF then fuses two independently-computed guesses, which is a reasonable approximation but still a guess — it can't catch a chunk that's the right answer for subtle reasons neither BM25's keyword overlap nor cosine similarity would surface.

So retrieve wider than you need (e.g. top 20, instead of the top 2-3 that'll actually go in the prompt), then rerank that shortlist with a slower, more accurate model that looks at the query and each candidate together. Reading them jointly is strictly more informative than reading them separately — the tradeoff is that it can't be precomputed, so it only makes sense on a small shortlist, never the whole corpus.

How a bi-encoder actually works

A bi-encoder is a transformer — the same architecture family as the chat model, but trained to output an embedding instead of generated text — run twice, on two completely separate inputs, with no path for information to cross between the two runs:

Two independent forward passes
q_vec = Encoder(query)
d_vec = Encoder(document)
score = cosine_similarity(q_vec, d_vec)
  • The query's tokens are turned into embeddings, passed through the transformer's self-attention layers, and pooled (usually the [CLS] token's final hidden state, or a mean over all token positions) into one fixed-size vector, q_vec.
  • The document goes through the exact same encoder, independently, producing d_vec.
  • Inside the transformer, self-attention lets query tokens attend to other query tokens — it never sees the document at all, and vice versa.
  • The two vectors only ever meet afterward, outside the model, as a single dot product.
  • Why caching works: Encoder(document) doesn't depend on the query in any way, so nothing about it needs to be recomputed when a new query arrives — document vectors can be computed once, offline, and reused for every future query. This is exactly what embed() in the fundamentals notebook does, and what section 02's vector search here is built on.

How a cross-encoder actually works

A cross-encoder is the same transformer architecture, but given a single concatenated input instead of two separate ones:

One joint forward pass
input = [CLS] query [SEP] document [SEP]
score = ClassifierHead( Encoder(input) )
  • Every token from the query and every token from the document now sit inside the same sequence, so every self-attention layer computes attention scores between all token pairs — a query token can directly attend to a specific document token, weigh how much it matters, and pass that interaction forward into deeper layers.
  • The [CLS] token's final hidden state (now shaped by attending across the whole query-document pair) gets fed into a small classifier head that outputs a single relevance score.
  • Why it's more expressive: a bi-encoder is mathematically forced to compress all of a document's meaning into one fixed vector before it ever knows what question will be asked of it; a cross-encoder gets to look at the document in light of the specific query, token by token, before deciding anything.
  • Why it's slow: Encoder(input) takes the query as part of its input, so this forward pass cannot be run until the query exists, and cannot be cached for reuse against a different query — the cost is exactly what breaks precomputation.
Bi-encoder (fast, independent) vs. cross-encoder (slow, joint)
One-line takeaway
Separate passes + late dot-product = fast but blind to how query and document actually relate. One joint pass with shared attention = slow but catches relationships (negation, indirect phrasing, outright contradiction) a vector comparison structurally cannot see.
04

Chunking strategy tradeoffs

  • Every technique so far assumes the corpus is already split into reasonably-sized chunks — but how it got split determines what's even possible to retrieve.
  • A chunk is the unit retrieval operates on: if a fact spans a chunk boundary, no amount of clever hybrid search or reranking can retrieve it whole, because it was never stored whole.
  • The fundamentals notebook used fixed-size character chunking, which is simple but blind to sentence and paragraph structure. In production this is a real tuning knob, not an afterthought — the same source paragraph splits very differently depending on strategy, and where the boundary lands decides what survives retrieval intact:
Same paragraph, three chunking strategies

Overlap (e.g. 10-20% of chunk size) stops a fact from landing exactly on a chunk boundary and becoming unretrievable from either side — visible above as the amber shared region between consecutive chunks. Too much overlap wastes storage and duplicates results; too little risks losing facts that straddle a cut.

05

Evaluation — retrieval metrics

  • Everything in sections 02-04 was a design choice — BM25 weight vs vector weight, chunk size, whether to rerank. Design choices need a number attached to them, or "we changed the chunking strategy" is just a guess about whether things got better.
  • That number comes from building a small hand-labeled eval set: a list of (query, known-relevant chunk) pairs where a human has already decided what the "correct" retrieval result is.
  • Every metric below runs the retriever against every query in that eval set and scores how well the ranked list it returned matches what should have come back.

If you've done any classification work, Precision and Recall aren't new — they're the same TP / FP / FN / TN confusion matrix from binary classification, just applied to "was this chunk retrieved" instead of "did the model predict positive." The mapping is exact:

Confusion matrix, applied to retrieval — 6 chunks, 1 query
  • Two independent boundaries drawn over the same 6 chunks — ground truth (is it actually relevant to this query?) and system output (did the retriever put it in the top k?) — and every chunk falls into exactly one of four boxes depending on which side of each boundary it's on.
  • TN is normally the biggest bucket by far (most of the corpus is irrelevant to any given query and correctly left out), which is exactly why it's dropped from both formulas below — a metric that rewarded "correctly ignored" would be nearly 1.0 for almost any retriever and say nothing useful.

Precision@k — of what you handed back, how much was actually relevant?

  • Imagine a retriever that, paranoid about missing anything, just returns every chunk in the corpus for every query. It would never miss the relevant one — perfect recall — but the prompt would be drowned in noise.
  • Precision is the metric that punishes exactly that: it only cares about the purity of what came back, not whether the right thing was in there somewhere.
Precision at k
Precision@k = TPTP + FP = relevant chunks in top kk
  • TP — relevant chunks the system retrieved. FP — irrelevant chunks it retrieved anyway.
  • TP + FP is just k — every retrieved chunk is either a true positive or a false positive, so the denominator is always the size of the retrieved set.
  • Averaged across every query in the eval set to get one final Precision@k number for the system.

Retrieving 1 relevant chunk inside a top-3 list scores 1/3 — even when that one chunk is exactly the right one — because the other two slots were wasted. That's the tradeoff Precision exposes: a small, tight top_k naturally scores higher on precision than a wide one.

Recall@k — of what's relevant, how much did you actually retrieve?

  • Now flip the question. A retriever that returns exactly one, extremely precise chunk per query looks great on precision — but if that one chunk is the wrong one, the user gets no answer at all.
  • Recall is the check on precision: did the system find the relevant chunk somewhere in what it returned, regardless of how much else came along with it?
Recall at k
Recall@k = TPTP + FN
  • TP — relevant chunks retrieved. FN — relevant chunks that existed but were missed (not in the top k).
  • TP + FN is the total count of chunks that were actually relevant to the query — the denominator doesn't depend on k at all, unlike Precision's.
  • In our eval set (one known-relevant chunk per query) this collapses to the simple binary case: found it → 1/(1+0)=1, missed it → 0/(0+1)=0. Averaged across the eval set, so Recall@3 = 0.667 means "found in the top 3 for two-thirds of the queries tested."
  • Precision and Recall are in direct tension: widening top_k can only help or hold Recall steady (more chances to include the right chunk) while it can only hurt or hold Precision steady (more chances to include noise).
  • Neither number alone tells the full story — that's why production systems track both, not one.

MRR — Recall says "found it," but found it where?

  • Recall@3 treats a hit at rank 1 and a hit at rank 3 as the same success. But they're not the same to the generation step downstream: section 07 shows that a chunk buried in the middle of a stuffed context window gets less attention from the LLM than one at the front.
  • So a retrieval metric that rewards landing near the top, not just landing somewhere in the list, is measuring something Recall can't see.
Mean Reciprocal Rank
MRR = average of 1rank of first relevant result
  • Rank 1 → contributes 1/1 = 1.0. Rank 2 → 1/2 = 0.5. Rank 5 → 1/5 = 0.2 — a steep, deliberate penalty for landing further down.
  • A query where nothing relevant was found in the retrieved list contributes 0.
  • "Mean" — average this reciprocal rank across every query in the eval set.

NDCG@k — MRR, generalized past "there's exactly one right answer"

  • MRR only knows about the first relevant hit, and only knows "relevant" as yes/no. Real eval sets are often messier: a query might have two or three acceptable chunks, and some might be a better answer than others (a chunk that directly answers the question vs. one that's merely on-topic).
  • NDCG — Normalized Discounted Cumulative Gain — handles both: it sums up every relevant hit in the list (not just the first), discounts each one by how far down the list it landed, and then normalizes against the best possible ordering so the number is comparable across queries even when they have different numbers of relevant chunks.
Discounted Cumulative Gain at k
DCG@k = Σi=1..k relevanceilog₂(i + 1)
  • relevancei — how relevant the result at position i is (0/1 for binary relevance, or a graded score like 0-3 for "a little / very relevant").
  • log₂(i + 1) — the discount. At i=1 it's log₂2=1 (no discount); by i=3 it's log₂4=2 (half credit) — later positions count for steadily less, but never hit zero.
Normalized DCG at k
NDCG@k = DCG@kIDCG@k
  • IDCG@k — the DCG of the ideal ordering: same relevant chunks, but sorted so the most relevant one comes first. This is the best score physically possible for that query.
  • Dividing by it turns a raw, corpus-dependent DCG number into a 0-1 score that means the same thing — "how close to the best possible ordering" — for every query, regardless of how many relevant chunks it happened to have.
Worked example — 3 queries, top-3 retrieved each
06

Evaluation — generation metrics

  • Section 05's metrics can all score perfectly — the exact right chunk retrieved at rank 1 — and the generated answer can still be wrong.
  • Retrieval only checks whether the right raw material made it into the prompt; it says nothing about what the model actually did with that material once it got there.
  • That's a second, independent failure surface, and it needs its own metrics.

Faithfulness / groundedness — did the answer stick to the source?

  • An LLM generating from context can still drift — filling gaps with what sounds plausible from its training data rather than what the retrieved text actually says.
  • Faithfulness catches this by breaking the answer into its individual factual claims and checking each one, independently, against the context: is this specific claim actually stated (or directly implied) there, or did the model add it?
Faithfulness
Faithfulness = claims supported by the contexttotal claims in the answer
  • "Supported" is usually judged by a second LLM call acting as a grader — shown the context and one claim at a time, asked a strict yes/no.
  • A score below 1.0 means at least one claim in the answer isn't backed by anything that was retrieved — the definition of a hallucination in the RAG setting.

Answer relevancy — grounded is necessary, but is it also on-topic?

  • Faithfulness alone has a blind spot: an answer can be built entirely from real sentences in the context and still not address what was asked — for instance quoting a true, context-backed fact that has nothing to do with the question.
  • Answer relevancy checks the other direction: does this answer actually respond to this query? The trick is there's no direct way to score "answers this question" — so it's measured indirectly, by working backwards.
Answer relevancy Answer relevancy = average ( cosine_sim ( embed(original query), embed(generated questioni) ) )
  • An LLM reads the answer and generates n plausible questions it would be a good response to — reverse-engineering "what was this answer trying to answer?"
  • Each generated question is embedded and compared to the original query's embedding with cosine similarity (the same operation from RAG Fundamentals section 03) — high similarity means the answer's implied question matches the one actually asked.
  • Averaged over the n generated questions into one score per answer.

Hallucination rate — faithfulness, rolled up across a whole eval set

  • A single faithfulness score describes one answer. Production needs a number that describes the system over time — is quality holding steady, or slipping?
  • Pick a faithfulness threshold (commonly "must be 1.0, fully supported"), run it over every answer in an eval set, and count how often it fails.
Hallucination rate
Hallucination rate = answers failing the faithfulness thresholdtotal answers evaluated
  • Tracked over time like any other production metric — a sudden jump with retrieval metrics unchanged (section 05) points at the prompt template or context handling, not retrieval, since retrieval quality didn't move.

LLM-as-judge — the mechanism behind every metric above, made explicit

  • Faithfulness, answer relevancy, and hallucination rate all quietly rely on the same trick: instead of computing a score with a formula, hand the question to a second LLM call and ask it to judge.
  • "Is this claim supported by this context?" is not something you can compute with arithmetic — it requires reading comprehension.
  • LLM-as-judge is the general pattern name for that: a grading prompt, a rubric, and a model call that returns a score instead of an answer.
The general shape judge_score = LLM( grading_prompt ⊕ rubric ⊕ (question, answer, context) )
  • Strength: no reference answer required, handles open-ended questions, catches semantic issues (contradiction, unsupported claims) that word-overlap metrics structurally can't see.
  • Weakness: costs an API call per eval (real money, real latency, at eval-set scale), inherits the judge model's own blind spots and biases, and is only as reliable as the rubric — a vague grading prompt produces a vague, inconsistent judge.
  • Practical mitigation: use a strict rubric with few-shot examples, prefer structured output (a number or enum, not free text) so scores are comparable run to run, and spot-check the judge against human ratings occasionally rather than trusting it blindly forever.

Classical NLG metrics (BLEU, ROUGE, METEOR) — only when a reference answer exists

  • Before LLM-as-judge, machine translation and summarization used n-gram overlap metrics: given a reference answer written by a human, count how many words or word-sequences the generated answer shares with it.
  • They still show up in RAG evals when a gold reference answer happens to exist for every query — but that's a real precondition, and RAG question-answering usually doesn't have one, because there are many equally correct ways to phrase the same true answer.
ROUGE-N (recall-oriented, the common RAG-adjacent variant)
ROUGE-N = matching n-grams (generated ∩ reference)total n-grams in the reference
  • BLEU — precision-oriented sibling of ROUGE (matching n-grams over total n-grams in the generated text, with a penalty for overly short outputs). Born for machine translation, where there's exactly one reference sentence to match.
  • METEOR — extends the idea with stemming and synonym matching, so "refund" and "reimbursement" can partially match instead of counting as a total miss — an improvement, but still fundamentally counting shared words, not verifying facts.
  • Why they're a weaker fit for RAG: "refunds are accepted within 60 days" and "you get 60 days to request a refund" share almost no n-grams and would score low, despite being the same correct answer — the exact failure mode faithfulness (LLM-as-judge) doesn't have, since it reads for meaning, not surface overlap.

07

Failure modes

  • Every section so far fixed one specific weak point. But a production RAG system fails in a handful of recurring shapes that don't map cleanly to any single component — they emerge from how retrieval and generation interact.
  • Recognizing the shape of a failure is what tells you which section above actually fixes it.
Lost-in-the-middle — attention weight by position in context
Stale index
  • The embedding stored for a chunk is a snapshot of that text at the moment it was indexed — it isn't linked to the live document in any way that would notice a later edit.
  • So a source document changes, the vector store still holds the old embedding pointing at old text, and the retriever confidently returns outdated information with no error, no warning, nothing that would tell you it happened.
  • This is exactly the freshness problem section 09 solves — a retrieval system is only as current as its last (re-)index.
Hallucinated citations
  • The model cites a source that either wasn't retrieved or doesn't say what the citation claims.
  • This happens because "cite your sources" is an instruction the model follows the same way it follows any other instruction — by generating plausible-looking text — and a plausible-looking citation isn't the same thing as a real one.
  • Faithfulness checking (section 06) catches this at the answer level by verifying every claim against what was actually retrieved, not by trusting the citation itself.
Retrieval-generation mismatch
  • The most frustrating failure mode, because retrieval did its job — the right chunk is sitting right there in the context — and the answer is still wrong.
  • The break happens downstream: the prompt template buries the chunk among too many others, truncates it before the key sentence, or the model simply defaults to its own parametric knowledge instead of the provided context.
  • This is why retrieval metrics (section 05) and generation metrics (section 06) have to be tracked as two separate numbers — a good Recall@k with a bad faithfulness score is the fingerprint of exactly this failure.
08

Scaling & infra

  • Every retriever in this tutorial so far — the fundamentals notebook's VectorStore, section 02's hybrid search — compares the query against every single stored vector, every query. That's O(n): double the corpus, double the work per query.
  • Fine for a demo, unworkable past roughly 10k–100k vectors depending on latency budget, because query time keeps growing in lockstep with how much data has been indexed — a problem no clever ranking formula fixes, since the issue is the number of comparisons, not how they're scored.
  • The fix is to stop comparing against everything, using structure instead of brute force to skip most of the corpus per query.
  • HNSW (Hierarchical Navigable Small World) builds a multi-layer graph over the vectors at index time: the top layer has only a few nodes with long-range links, letting a query jump close to the right neighborhood in just a few hops; each layer down adds more nodes with shorter, denser links for progressively finer-grained search — like knowing a city's highway system gets you to the right district in a couple of turns, and only then do you need street-level directions.
  • That's the intuition behind why query time grows with log n instead of n — most of the corpus is never touched, because the graph already skipped it.
Query time, brute-force vs HNSW, as corpus grows

Where the vectors live

  • pgvector — already running Postgres? Add the extension, keep vectors next to relational data, one less system to operate.
  • Chroma / Qdrant — purpose-built, easy to self-host, good default for a mid-size app.
  • Pinecone / Weaviate Cloud — managed, scales without operating the index, costs more per GB.

Cost/latency knobs, roughly in order of impact

  1. top_k — every extra retrieved chunk adds prompt tokens (cost) and dilutes context (quality).
  2. Rerank depth — reranking 50 candidates instead of 10 multiplies reranker cost ~5x for often marginal gains past the top ~20.
  3. Embedding dimensionality — a 3072-dim embedding costs more to store and search than 768-dim; many providers offer truncated/quantized variants (e.g. Matryoshka embeddings) for a small quality hit.
09

Freshness & updates

  • Step back and every section in this tutorial has made one assumption: the corpus is fixed at index time. Section 07's "stale index" failure is what happens when that assumption quietly breaks — real documents get edited, deleted, replaced, and the index has no built-in way to notice.
  • An index is best understood as a cache of your documents as they were at embedding time, and like any cache, it needs an explicit invalidation strategy or it silently drifts from the truth.
  • The key piece that makes invalidation possible at all is a stable chunk_id (e.g. a hash of source_doc_id + chunk_offset) — without one, there's no way to know that a newly-embedded chunk is an update to something already in the store rather than a brand new entry, so a naive re-embed just piles up duplicates instead of replacing anything.
Upsert store — same id in, old vector replaced, no duplicate
  • Point-in-time consistency is the sharpest edge of this: if a document updates while a batch of queries is mid-flight, some queries can land on the old vector and some on the new one — the store has no single moment where "the update happened" for every reader at once.
  • Chasing that down with distributed transactions is usually not worth it; the pragmatic answer is to accept eventual consistency and just keep the update-to-visible lag small enough that it rarely matters.
