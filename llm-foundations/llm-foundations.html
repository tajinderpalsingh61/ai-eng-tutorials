LLM Foundations

Talk-through + live API calls
01

What is a language model, really

Strip away the hype: an LLM is a function that takes a sequence of tokens and predicts the most likely next token, one at a time, repeatedly.

The core loop

"The capital of France is" → model scores every possible next token → picks (or samples) one → appends it → repeats.

Everything — chat, code generation, agents — is this loop wrapped in formatting and tooling. There is no separate "understanding" step; prediction is the mechanism.

02

Tokens, not words

Models don't see words. They see tokens — chunks of text, often sub-word pieces.

  • "unbelievable" might split into un + believ + able
  • Common words are usually one token; rare words, code, and non-English text use more tokens
  • This is why pricing and context limits are measured in tokens, not characters or words

Why this matters for engineering

Cost, latency, and context budget are all token-driven. A prompt that "looks short" in a foreign language or in code can burn far more tokens than the same idea in plain English.

02b

How tokens get made — Byte Pair Encoding

The idea

Start with individual characters. Repeatedly find the most frequent adjacent pair in the training text and merge it into a single new token. Repeat thousands of times until you hit a target vocabulary size (e.g. ~100k tokens).

Result: frequent whole words end up as one token ("the"), rare or foreign words get split into smaller familiar pieces ("unbelievable" → un+believ+able) — which is exactly the token-cost behavior from the previous section.

One-line takeaway
BPE is just frequency-based compression of text into a fixed vocabulary — it's why token boundaries look arbitrary but aren't random.
03

Attention, at the intuition level

This is the one mechanism worth understanding clearly — everything else in AI engineering (RAG, context management, cost) traces back to it.

The idea in one sentence

For every token, the model asks "which other tokens in this context actually matter for understanding me?" — and weighs them accordingly, instead of just reading left-to-right blindly.

Older models (RNNs) processed text strictly left to right and lost track of anything far back. Attention lets every token look at every other token directly, regardless of distance — that's the unlock.

Two consequences worth internalizing

  1. Why not just paste the whole document into the prompt? — Attention computes a relationship between every pair of tokens. Double the input length, and the compute roughly quadruples. This is precisely why RAG exists: retrieve only the relevant chunks instead of paying for full-document attention every time.
  2. Why does a long conversation get slower / costlier? — Every new token re-attends to the full growing context. Systems cache the earlier computation (KV cache) so it isn't redone from scratch — this is why the first token takes a moment ("time to first token") but generation speeds up after.

Attention matrix — why it's O(n²)

Every token must score against every other token. n tokens → n × n pairs → attention matrix size grows as n², not n. Double context length, compute (and memory for that matrix) roughly quadruples, not doubles.

Bytes = n² × 2 (fp16), one attention head — real models have dozens of heads and layers, so actual memory is this number × heads × layers. Length up 128×, matrix up ~16,000×. That's why "just paste the whole codebase in" breaks down long before the token limit does — the model isn't scaling linearly, it's scaling quadratically, and so is your bill.
04

Context window — the budget you're always managing

The context window is the total tokens (input + output) a model can attend over at once.

Small context

Cheaper, faster, forces you to be selective about what you feed the model — pushes toward RAG, summarization, chunking.

Large context

Tempting to "just dump everything in," but quality often degrades on info buried in the middle ("lost in the middle") and cost still scales up.

Practical framing for engineers: the context window is not free real estate — every token you add is a tradeoff against cost, latency, and the model's ability to actually use it.

05

Prompting basics

System vs. user role

system sets standing behavior ("You are a precise, concise coding assistant"). user is the actual request. Keep system prompts stable; vary user turns.

Zero-shot vs. few-shot

Zero-shot: just ask. Few-shot: show 2–3 examples of input→output before the real request — dramatically improves consistency of format/style for tasks like classification or extraction.

05b

Sampling parameters

The model's forward pass ends in a softmax over the vocabulary — a probability for every possible next token. Sampling parameters decide how that distribution turns into one chosen token. Same distribution, three different knobs, three different outcomes. Play with the lab below — it's computing a real softmax, not canned numbers.

Sampling lab — prompt: "The weather today is ___"
Move the slider or pick a filter, then draw a sample.

temperature — reshapes the distribution before sampling

Temperature divides the logits by T before softmax, then renormalizes. It doesn't remove tokens — it changes how peaked or flat the whole curve is. Drag the slider above to 0.3 and watch "sunny" swallow almost everything; drag it to 1.8 and watch the tail (stormy, dry) become genuinely reachable.

Low temperature for extraction/classification (you want the single best answer, repeatably). Higher for brainstorming/creative writing (you want the tail to be reachable).

Final pick: temperature never removes candidates — every token stays eligible, just reshaped. The draw is a weighted random pick straight from this reshaped distribution. At T → 0 it's so spiked the draw always lands on the top token — effectively argmax.

top_k — keep a fixed number of candidates

Sort tokens by probability, keep only the top k, drop the rest before drawing. Simple and predictable, but blind to shape: k=3 always keeps exactly 3 tokens, whether the distribution is sharply peaked or nearly flat. Click "top_k = 3" above — the cut tokens gray out and become undrawable.

Final pick: the survivors get renormalized to sum to 1, then one is picked via weighted random draw over just those — click "draw sample" to watch it happen with a live cumulative-range readout.

top_p (nucleus sampling) — keep enough tokens to cover a probability mass

Sort tokens by probability, walk down accumulating a running cumulative sum, stop as soon as it reaches p. The candidate set size adapts to the shape of the distribution — a confident/peaked distribution needs few tokens to hit p; a flat/uncertain one needs many. Click "top_p = 0.8" above and compare which tokens survive versus top_k=3 — nucleus sampling can keep more or fewer tokens depending on how the mass is distributed.

Final pick: same finish as top_k — renormalize the survivors, then weighted random draw.

max_tokens & stop_sequences

max_tokens caps output length (hard cutoff — can truncate mid-sentence). stop_sequences tell the model to stop generating the moment it produces a given string — useful for structured formats.

05c

Getting structured output

In production you rarely want free-text back — you want JSON you can parse and pass to the next system.

Approach 1: prompt for JSON directly

Explicitly instruct the format and give a schema/example. Works, but the model can still occasionally add stray text or break formatting.

Approach 2: tool/function calling (more reliable)

Define a "tool" with a strict input schema. The model is constrained to return arguments matching that schema — this is the production-grade way to get structured data, and it's the same mechanism agent tool-calling is built on.

Prompt caching (cache_control)

A repeated prefix — long system prompt, few-shot examples, a big doc — costs full tokens every call unless you cache it. Provider stores the prefix's computed state once; next call that shares it pays a much cheaper "cache read" instead of full price. Step through one prefix across two calls:

Gemini — explicit cache object

You create a CachedContent via client.caches.create(...) and get a handle back (cache.name). Every later call passes cached_content=cache.name to reuse it. You own the object — its TTL, its lifetime, its deletion.

Anthropic — inline flag

You mark a content block itself: "cache_control": {"type": "ephemeral"}. No separate object — the provider caches automatically and reuses it on the next call that shares that exact prefix, within a rolling TTL (default 5 min). Response usage reports cache_creation_input_tokens vs cache_read_input_tokens so you can see the hit.

When it's worth it

Only pays off once the shared prefix is reused across calls — a single one-off prompt gains nothing (cache write itself often costs slightly more than an uncached call). Best fits: chat with a stable long system prompt, agent loops re-sending the same tool definitions/instructions, RAG with a large fixed document.

05f

LiteLLM vs LangChain — why not just call the provider SDK?

Calling the provider SDK directly is fine for a single-provider prototype. It becomes a liability once you need to swap providers, add a fallback model, or track cost/usage centrally — each SDK has its own call shape, message format, and error types.

LiteLLM — thin unified call layer

One function, litellm.completion(...). Swap the model= string ("gemini/gemini-2.5-flash-lite", "claude-sonnet-5", "gpt-4o") and the call shape and response shape stay identical. Minimal magic — closest thing to a raw SDK that's still provider-agnostic. Built-in retry/fallback/cost-tracking hooks.

LangChain — same unification, plus orchestration

init_chat_model(...) gives the same .invoke() interface across providers, but the framework's real value is chains/agents/memory/tool-orchestration built on top. Worth the extra weight once you need those — overkill for a single one-off call.

Rule of thumb

Raw SDK → fine for a prototype tied to one provider. LiteLLM → default choice once you need multi-provider or fallback support without extra framework weight. LangChain → once you're building multi-step chains or agents, not just single calls.

05g

Real-world best practices for LLM calls

Prototype code calls the model and hopes for the best. Production code has to survive rate limits, transient failures, and models that occasionally don't follow instructions.

Retries + backoff

Transient 429/5xx responses need exponential backoff and a retry, not an immediate fail — a single blip shouldn't surface as a user-facing error.

Timeouts

Always set an explicit timeout on every call. Without one, a hung request can block a worker indefinitely.

Error handling

Catch rate-limit, auth, and content-filter errors distinctly — each needs a different response (backoff vs. fail-fast vs. flag-for-review). A blanket except Exception hides which one happened.

Rate-limit handling

Respect the provider's Retry-After header, and queue or shed load rather than hammering the API when you're being throttled.

Structured output validation

Validate the model's JSON/tool-call output against a schema (e.g. Pydantic) before trusting it downstream — models occasionally violate schema even with tool-calling enabled.

Logging / observability

Log prompt, response, token usage, and latency per call (redact sensitive data) — without this, failures, cost spikes, and quality regressions are undebuggable after the fact.
